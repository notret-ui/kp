# Спецификация: Генератор КП «Печь.ру»

Дата: 2026-06-07

## 1. Цель

Инструмент для менеджеров по продаже печей и каминов (pech.ru): собрать красивое
коммерческое предложение (КП) из выбранных товаров и выдать клиенту **веб-ссылку** и **PDF**.
Визуал — пиксельно по Figma-шаблону «КП-200825-ДАО» (см. `docs/design-system.md`).

Локально на Маке менеджера; архитектура готова к переезду на сервер для команды.

## 2. Рабочий цикл (выбран пользователем — «цикл Атлантов»)

1. **Концепция и структура** — здесь, в Claude Code (эта спека + дизайн-система + структура репо).
2. **GitHub** — заливаем репозиторий.
3. **Claude design** — даём доступ к репо, собираем **прототип** (HTML/CSS-мокапы всех слайдов
   на основе `docs/design-system.md`). Сначала мокапы → апрув → дальше.
4. **Claude Code** — реализация: бэкенд + данные подключаются к утверждённым мокапам;
   все обновления через коммиты на GitHub.

> Принцип Руслана: сначала дизайн-система, потом мокапы, потом код. У нас дизайн-система
> уже задана Figma — мы её извлекаем, а не выдумываем.

## 3. Модель КП (динамический набор слайдов)

Порядок:

1. **Обложка** — оффер + 6 преимуществ + **имя клиента и дата** (приветствие). *Всегда.*
2. **О компании** «с 2009 г.». *Всегда.*
3. **Карточка товара** — 1 слайд на каждую выбранную позицию. *Динамически (N штук).*
4. **Итоги** — таблица позиций, доставка/монтаж/услуги отдельными строками,
   скидка на КП, ИТОГО. *Всегда, если позиций ≥ 1.*
5. **Галерея** «Фото довольных клиентов». *Опционально (галочка).*
6. **Товары в этой категории** — авто-подбор сопутствующих. *Опционально.*
7. **Контакты + менеджер**. *Всегда.*

Длина КП = фикс-слайды + по одному на товар.

## 4. Данные (гибрид)

- **YML-фид pech.ru** — структура, цена, РРЦ (для зачёркнутой цены), фото, параметры.
  Переиспользуем рабочую интеграцию из `Parsing/Kamindom/config.py`:
  - `PECH_XML_URL` — основной фид; `PECH_SKLADE_XML_URL` — фид «на складе»
    (`price` = со скидкой, `oldprice` = РРЦ).
  - `BRAND_DISCOUNTS` (напр. Астов −20%), `S3_BUCKET`/`S3_ENDPOINT`.
  - **Идентификатор товара = `offer id`** (артикула в фиде нет).
  - Секрет `APIFY_TOKEN` в том файле — не копировать в репо, читать из env/.env (gitignore).
- **Сайт pech.ru** — доп. фото + длинное описание; **кэш в SQLite** (один заход на товар).
- **SQLite + FTS** — индекс по всему фиду: мгновенный поиск по offer id / названию / бренду /
  категории.

## 5. Поток менеджера

Веб-форма: поиск и добавление товаров → кол-во, индивидуальные скидки → строки
доставки/монтажа/услуг → имя клиента, контакты менеджера, общая скидка на КП →
«Сгенерировать». КП сохраняется в SQLite (переоткрытие/редактирование).
Выдача: `/kp/<id>` (веб) и `/kp/<id>.pdf` (PDF).

## 6. Архитектура

Python. Один Jinja2-шаблон → и веб-страница, и PDF (через Playwright/headless Chromium).

Модули (чёткие границы, тестируются независимо):

- `feed/` — загрузка/парсинг YML, построение SQLite-индекса. Зависит от: сеть/локальные XML.
- `scraper/` — добор фото/описаний с сайта + кэш. Зависит от: сеть, SQLite-кэш.
- `catalog/` — поиск по товарам (SQLite FTS). Вход: запрос → выход: список товаров.
- `kp/` — модель КП, сохранение, бизнес-логика (суммы, скидки, итоги). Чистая логика.
- `render/` — Jinja2-шаблоны слайдов + Playwright→PDF. Вход: модель КП → выход: HTML/PDF.
- `web/` — FastAPI: форма, `/kp/<id>`, `/kp/<id>.pdf`.

Стек: **FastAPI + Jinja2 + Playwright + SQLite (FTS5)**.

## 8. Границы (YAGNI)

- Не делаем: авторизацию/мультипользовательность (пока локально), редактор дизайна,
  интеграции с CRM. Закладываем структуру, добавим при переезде на сервер.
- Прототип (мокапы) делается в Claude design на статике; данные не трогаем до апрува визуала.

## 9. Открытые пункты для сверки

- Точные Figma-переменные (цвета/шрифты) — сверить через `get_variable_defs` после снятия лимита.
- Гарнитура шрифта.
- Состав строк услуг в слайде «Итоги» — финализировать на этапе мокапов.
